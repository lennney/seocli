# 迭代评审模式

基于gamixo项目实践：方案经历了3次重大方向调整才找到可行路径。

## 迭代循环（4轮制）

```
Round 0: 方向探索（当原方向被推翻时）
  ↓ 并行调研 + 用户手动验证
Round 1: 独立评审（4角色打分）
  ↓ 发现致命问题
Round 1.5: 调研（上网搜解决方案/行业数据）
  ↓ 方向可能需要调整
Round 2: 辩驳+解决方案（角色读取他人意见后反驳）
  ↓ 达成共识
Round 2.5: 假设验证（设计MVT测试核心假设）
  ↓ 用真实数据验证
Round 3: 行动计划 + 交付物
  ↓ PRD（先出，用户确认）
  ↓ IA + HTML原型（确认后）
  ↓ 技术方案 + 运营SOP
```

## gamixo实际迭代路径

```
v1: 泛游戏资讯+攻略（RSS聚合+AI改写）
  → 评审结论：内容农场风险、导流假设不成立
  → 调研结论：必须改为AI+人工模式

v2: H5游戏攻略站
  → 验证结论：搜索量太小，"H5游戏攻略"月搜索约200

v3: 每日谜题游戏+攻略内容
  → 用户手动搜索验证："strands hints today"有8万月搜索
  → 调研结论：模式已被NYT验证
  → 方向确认
```

## 关键教训

1. **不要在第一个方案上死磕**：评审发现问题后，要敢于推翻重来
2. **用真实数据验证假设**：不要猜"有没有人搜"，直接去搜
3. **用户参与验证**：让用户手动搜索关键词、查看竞品，比AI编数据可靠
4. **每次迭代缩小范围**：从泛→窄，最终找到可行的细分切入点
5. **评审+调研+验证是三位一体**：评审找问题，调研找方案，验证确认可行性
