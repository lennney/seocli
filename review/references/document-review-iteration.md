# 文档迭代评审模式（评审→修复→再评审）

方向确认、文档产出后，还需要对文档本身做多轮评审。
这是独立于"方向探索迭代"的第二层迭代。

## 实际案例：gamixo Daily 3轮文档评审

```
第一轮（3个角色独立评审）：
  UX: 3.0 / 技术: 3.5 / 运营: 2.5 → 综合 3.0
  发现22个问题（5个P0、7个P1）
  → 修复16项

第二轮（验证修复+发现新问题）：
  UX: 3.2 / 技术: 4.0 / 运营: 3.5 → 综合 3.6
  第一轮修复率95%，但发现17个新问题（文档间不一致为主）
  → 修复全部

第三轮（最终验证）：
  UX: 3.6 / 技术: 4.2 / 运营: 3.8 → 综合 3.9
  问题更细、更少（2个阻塞、4个高优、9个中优）
  → 修复16项
```

## 每轮评审角色分配建议

```
第一轮（找大问题）：UX + 技术 + 运营（3角色并行）
  → 重点：架构问题、可行性问题、排期问题

第二轮（验证+找细节）：同上角色重新评审
  → 重点：修复是否到位、文档间一致性、新发现的问题

第三轮（最终确认）：可以合并为2个角色（UX+技术双重角色、运营）
  → 重点：阻塞级问题、原型与文档对齐度
```

## PM文档产出顺序

用户偏好：先出PRD，确认后再做其他的。

```
第一步：PRD（产品需求文档）
  → 用户确认后才做后续文档

第二步：IA + HTML原型
  → 信息架构 + 可交互原型

第三步：用户故事 + Sprint排期
  → 开发团队直接使用的文档

第四步：竞品分析（可选但推荐）
  → 评审时经常被问"别人怎么做的"

每步之间都要经过团队评审迭代。
```

## 文档间一致性检查清单

每轮评审都要交叉检查以下项目：

```
□ URL格式：PRD ↔ IA ↔ 用户故事 ↔ 原型
□ 游戏/功能数量：PRD ↔ 用户故事 ↔ Sprint排期 ↔ 原型
□ 优先级：PRD ↔ 用户故事
□ 数字（词库规模、时间估算等）：PRD ↔ Sprint排期
□ 技术参数（ISR间隔、性能指标等）：PRD ↔ IA ↔ 技术方案
□ 品牌名/产品名：所有文档+原型
□ 战略方向（市场定位、语言等）：PRD ↔ 竞品分析
□ localStorage/状态管理格式：Schema文档 ↔ 原型代码 ↔ PRD
□ 时间估算（开发天数等）：PRD ↔ Sprint排期
□ 渲染策略（SSG/SSR/ISR参数）：PRD ↔ IA
```

## 评审中反复出现的问题类型

### 1. 文档间不一致（每轮都会发现新的）
修复方法：指定一个"权威文档"（通常是PRD），其他文档对齐。

### 2. 原型与文档脱节（文档改了原型没改）
修复方法：评审时用read_file读取原型代码验证，不要只读文档。

### 3. 排期与功能不匹配（P0功能排到Phase 2）
修复方法：交叉比对PRD功能优先级和Sprint排期中的任务。

### 4. "重开发轻运营"（开发时间充足，运营时间不足）
修复方法：运营角色评审时重点关注内容生产时间分配。

### 5. localStorage/前端状态schema不一致
修复方法：创建专门的schema文档，所有地方引用同一份。

### 6. 中英文混入（原型中混入非目标语言）
修复方法：评审时检查原型中的所有用户可见文本。

### 7. 旧方向文档残留（方向调整后旧文档未更新）
修复方法：方向确认后，标记所有旧文档为"留档参考"，新方向文档从头写。

## 评审评分跟踪模板

```markdown
| 维度 | 第一轮 | 第二轮 | 第三轮 | 趋势 |
|------|--------|--------|--------|------|
| UX   | 3.0    | 3.2    | 3.6    | ↑    |
| 技术 | 3.5    | 4.0    | 4.2    | ↑    |
| 运营 | 2.5    | 3.5    | 3.8    | ↑    |
| 综合 | 3.0    | 3.6    | 3.9    | ↑    |
```

## 何时停止迭代

- 没有阻塞级(Blocker)问题
- 高优(High)问题 ≤ 2个且有明确解决计划
- 文档间一致性检查全部通过
- 评分 ≥ 3.5/5
- 用户确认"可以进入开发"
